# 10 - 探索能力

## 目标

Somnia 的探索能力不是简单的“读文件工具集合”，而是要让 Agent 先建立项目地图，再决定读哪些文件、找哪些符号、如何验证文档与实际代码是否一致。

这一层能力主要解决三个问题：

1. 降低一上来就 `glob **/*` 或盲猜路径的倾向。
2. 让仓库结构、关键入口、符号位置有可复用记忆。
3. 让一次探索的结果能够沉淀到当前 session 和工作区级摘要中。

---

## 当前能力概览

### 1. 项目扫描

工具：`project_scan`

用途：

- 识别项目根目录
- 统计文件数、目录数
- 推断可能的技术栈
- 找出指导文件（如 `AGENTS.md`、`CLAUDE.md`、`README.md`）
- 找出清单文件（如 `pyproject.toml`、`package.json`、`go.mod`）
- 找出可能的源码根目录
- 找出可能的入口文件
- 输出浅层目录树

REPL 命令：

```text
/scan
/scan src
/scan --refresh src
```

说明：

- `/scan` 默认扫描当前工作区根目录。
- `/scan some/path` 只扫描指定子目录。
- 同一路径会优先命中 session 缓存。
- `/scan --refresh` 会忽略缓存并重新扫描。

### 2. 浅层目录树

工具：`tree`

用途：

- 在已经知道大致区域后，快速看一个目录的浅层结构
- 比大范围 `glob` 更适合建立局部心智地图

适合场景：

- 已经知道要看 `Runtime/`
- 想先看 `src/`、`Assets/`、`open_somnia/cli/` 这样的局部层次

### 3. 符号查找

工具：`find_symbol`

用途：

- 按符号名子串查找类型、类、接口、结构体、函数、方法
- 在读具体文件前先定位可能的实现位置

REPL 命令：

```text
/symbols
/symbols build
/symbols PaperComponent
```

当前交互方式：

1. 输入查询词
2. 弹窗列出匹配结果
3. 选择一个结果
4. 直接预览该位置附近源码

### 4. 文件与内容探索

相关工具：

- `glob`
- `grep`
- `read_file`

它们仍然是探索链路的一部分，但不再是推荐的第一步。

推荐顺序：

1. `project_scan`
2. `tree`
3. `find_symbol`
4. `glob` / `grep`
5. `read_file`

---

## 支持的语言/文件类型

当前 `find_symbol` 已支持的主要扩展名：

- `.py`
- `.cs`
- `.ts`
- `.tsx`
- `.js`
- `.jsx`
- `.java`
- `.go`
- `.rs`

说明：

- 当前实现是轻量正则扫描，不是完整语义索引。
- 目标是先解决“快速定位”问题，而不是一次做成全语言 LSP。

---

## 当前忽略策略

探索时会默认跳过一些高噪声目录，例如：

- `.git`
- `.open_somnia`
- `__pycache__`
- `.venv`
- `node_modules`
- `Library`
- `Temp`
- `Logs`
- `obj`
- `bin`
- `dist`

这能避免 Unity、Node、Python、构建产物等目录淹没有效结构。

## 持久化与记忆

### 1. Session 级探索缓存

位置：

```text
.open_somnia/sessions/<session_id>.json
```

字段：

```json
"exploration_cache": {
  "last_project_scan": {},
  "project_scans": [],
  "last_symbol_lookup": {},
  "symbol_queries": []
}
```

作用：

- 同一会话内复用之前的扫描结果
- 保留最近的符号查询历史
- 让 REPL 命令和后续推理共享探索上下文

### 2. Repo 级摘要沉淀

位置：

```text
.open_somnia/repo_summary.json
```

当前内容包括：

- 最近一次 `/scan` 的摘要文本
- 最近的符号查询摘要
- 更新时间

作用：

- 给后续 session 一个工作区级的探索起点
- 让系统提示词具备“已知仓库摘要”

### 3. 系统提示词注入

运行时会把两类探索记忆注入系统提示词：

1. `Repository memory`
2. `Session exploration memory`

原则：

- 这些内容只是提示，不是事实来源
- Agent 仍然要用工具再次验证再做强结论

REPL 命令：

```text

```

用途：

- 查看当前 investigation focus
- 查看最近读取的文件
- 查看最近扫描和搜索
- 查看已沉淀 facts
- 查看当前 exploration warning

## 设计取向

Somnia 当前选择的是“轻量、实用、可落地”的探索能力，而不是一开始就做重索引系统。

这意味着：

- 优先解决盲猜路径的问题
- 优先解决文档和代码结构不一致时的验证问题
- 优先解决一次探索结果无法复用的问题

暂不追求：

- 全仓库 AST 数据库
- 全语言语义索引
- 完整 cross-reference 图谱
- 一次性替代 LSP/IDE 的全部能力

---

## 已知边界

### 1. `find_symbol` 仍是正则级能力

当前优点：

- 快
- 依赖少
- 易跨平台

当前限制：

- 不是严格语义解析
- 对复杂语法、宏、生成代码、多行声明识别有限

### 2. `/symbols` 目前默认搜索路径是工作区根目录

当前实现更偏“全局找入口”，后续可以考虑：

- 允许先选路径范围
- 或支持 `/symbols path query` 这类形式

### 3. `/scan` 摘要仍然偏结构层

它现在更像“项目地图”，不是“架构理解报告”。

后续若要继续增强，可以补：

- 关键模块聚类
- 入口调用链摘要
- 主要领域对象摘要
- 测试覆盖面线索

---

## 后续建议

适合继续推进的方向：

1. 给 `/symbols` 增加路径范围选择。
2. 在 `/scan` 后支持继续下钻选择某个源码根目录并展开 `tree`。
3. 增加“探索日志压缩”策略，避免 session 缓存无限堆积。
4. 让 repo summary 从纯文本扩展到结构化字段，便于 UI 和后续提示词裁剪。
5. 针对 Unity、Python、Node 项目增加更强的栈特化提示。
