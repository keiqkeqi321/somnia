# 10 - 探索能力

## 当前定位

Somnia 当前只保留“即时探索”能力，不再内置探索结果持久化、repo summary、session 级 exploration cache，也不再提供 `/investigation` 面板。

探索阶段的原则：

1. 先建立局部项目地图。
2. 再定位符号和落点。
3. 最后读取最少量的必要源码。

---

## 当前可用能力

### 1. `/scan`

底层工具：`project_scan`

用途：

- 快速扫描当前仓库或某个子目录
- 输出项目根、文件/目录数量、可能技术栈、指导文件、源码根和浅层目录树
- 作为开始探索时的第一步，帮助 agent 建立项目地图

用法：

```text
/scan
/scan src
/scan --refresh src
```

说明：

- `/scan` 默认扫描工作区根目录。
- `/scan some/path` 扫描指定子目录。
- `--refresh` 目前只是兼容保留参数，不再涉及任何缓存刷新。
- `/scan` 每次都是即时查询，不会写入 repo summary，也不会复用旧扫描结果。

### 2. `/symbols`

底层工具：`find_symbol`

用途：

- 按名称子串查找类、函数、方法、接口等符号
- 在真正读文件前，先缩小实现范围
- 结合交互选择器快速预览命中的源码位置

用法：

```text
/symbols
/symbols build
/symbols PaperComponent
```

说明：

- 如果不带查询词，会弹出输入框。
- 找到多个结果后，会弹窗让用户选择一个结果进行源码预览。
- `/symbols` 也是即时查询，不会把结果沉淀成 session 记忆。

### 3. 其他探索工具

以下工具依然可供 agent 在运行时使用：

- `tree`
- `glob`
- `grep`
- `read_file`

推荐顺序：

1. `project_scan`
2. `tree`
3. `find_symbol`
4. `read_file`

---

## 为什么移除了探索记忆

上一版探索能力把扫描摘要、符号查询、文件读取等状态写进了 session 和工作区存储，还尝试把这些内容重新注入系统提示。

这套设计当前已经删除，原因是：

- 记忆质量不稳定，容易把过时结论重新带回后续轮次
- “读过”不等于“验证过”，会放大错误先验
- 探索输出和真正可复用知识没有被严格区分
- 记忆层和压缩层耦合后，排查行为退化更难定位

后续如果要做新记忆系统，会单独设计，不复用这套旧结构。

---

## 当前边界

- Somnia 现在不会持久化探索摘要。
- 不会维护 `.open_somnia/repo_summary.json`。
- 不会在 session 里写 `exploration_cache`。
- 不会提供 `/investigation`。
- 如果需要保留结论，应该在正常对话、任务系统或后续独立记忆系统里处理。
