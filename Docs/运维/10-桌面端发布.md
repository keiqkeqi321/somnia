# 10 - 桌面端发布

## 概览

本文件只说明 Somnia 桌面端的构建与发布。

当前桌面端发布包含：

- Windows 安装包：`.msi`
- macOS 安装镜像：`.dmg`

桌面端发布与 PyPI 发布目前共用同一套 tag 发布流水线：

- `.github/workflows/publish.yml`

另外保留一个手动内部构建入口：

- `.github/workflows/desktop-bundles.yml`

## 当前发布方式

正式发布由推送 `v*` tag 触发，桌面端相关 job 为：

1. `build-desktop`
2. `create-release`

`build-desktop` 负责构建桌面产物，`create-release` 负责把产物附加到 GitHub Release。

## 构建目标

当前矩阵构建目标为：

- `windows-latest` -> `x86_64-pc-windows-msvc` -> `msi`
- `macos-13` -> `x86_64-apple-darwin` -> `dmg`
- `macos-14` -> `aarch64-apple-darwin` -> `dmg`

上传到 GitHub Release 的是：

- Windows：`.msi`
- macOS：`.dmg`

## 发布触发

正式发布通常通过以下命令触发：

```powershell
powershell -File scripts\release.ps1 0.4.0
```

这会推送类似 `v0.4.0` 的 tag，并触发 `.github/workflows/publish.yml`。

如果只想手动验证桌面构建，可单独运行：

- `.github/workflows/desktop-bundles.yml`

## 当前实现状态

当前桌面构建是 unsigned build：

- Windows 使用 `--no-sign`
- macOS 还未接入 `codesign`
- macOS 还未接入 `notarization`

这意味着当前版本适合：

- 内部测试
- 开发预览
- 非正式分发

## Windows 签名说明

不做 Windows 签名，仍然可以安装，但有这些限制：

- `msi` 仍然可以生成
- 大多数机器上仍然可以安装
- 用户可能看到“未知发布者”或 SmartScreen 警告
- 某些企业环境可能直接拦截未签名安装包

建议：

- 内部测试版可以先不签名
- 面向外部用户的正式版应补 Windows 代码签名

## 构建依赖

桌面端构建链路依赖：

- Python
- Node.js / npm
- Rust / cargo
- Tauri CLI

Windows 本地脚本还会处理：

- workspace-local Rust toolchain
- `llvm-mingw`
- sidecar 打包

相关本地脚本：

- `scripts/release-desktop.ps1`
- `scripts/release-desktop.sh`
- `scripts/release/build_desktop_sidecar.py`

## Release 附件

当前 GitHub Release 会附加桌面端安装产物。

预期检查项：

- Release 已创建
- 存在 Windows `.msi`
- 存在 macOS x64 `.dmg`
- 存在 macOS arm64 `.dmg`

## Secrets

当前桌面端 unsigned 发布不强依赖额外签名 secrets。

未来如果补签名，需要增加：

- Windows 代码签名证书
- Windows 证书密码
- macOS codesign 证书
- Apple notarization 凭据

## 发布后验证

建议至少验证：

- GitHub Release 是否已创建
- Release 是否包含 `.msi` 和 `.dmg`
- Windows 安装包是否能安装并启动
- macOS 安装镜像是否可挂载并启动
- sidecar 是否随安装包正常运行

## 常见问题

### 1. Release 没有桌面附件

检查：

- `build-desktop` 是否成功
- `create-release` 是否成功
- bundle 输出目录下是否实际生成 `.msi` / `.dmg`

### 2. Windows 弹出未知发布者

原因：

- 当前未做 Windows 签名

处理：

- 内部测试可接受
- 外部正式分发前补代码签名

### 3. macOS 包可下载但启动受限

原因通常是：

- 未做 `codesign`
- 未做 `notarization`

处理：

- 当前仅作为 unsigned internal build 使用
- 正式外发前补签名与 notarization
