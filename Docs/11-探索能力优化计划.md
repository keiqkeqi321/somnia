# 11 - 探索能力优化计划

## 目标

这一轮优化的目标不是继续堆更多搜索工具，而是补齐 Somnia 的“探索控制层”，解决以下核心问题：

1. 反复读取同一段代码却没有新增证据。
2. 结论先于证据，过早宣称“已定位问题”。
3. 缺少事实、假设、排除过程的结构化沉淀。
4. 用户无法直接判断 agent 当前是在推进还是在打转。

---

## 分批计划

### 第一批：止损与可视化

目标：

- 建立结构化 investigation state
- 检测重复探索
- 给用户提供 investigation 面板

范围：

- 在 session 的 `exploration_cache` 中新增 investigation 字段
- 自动记录最近读取的文件、搜索过的符号、浅层目录探索
- 当同一路径被重复读取且没有新增事实时，生成 exploration warning
- 将 warning 注入系统提示词，阻止模型继续原地打转
- 增加 `/investigation` 命令查看当前探索状态

预期收益：

- 先止住“假推进”
- 让用户可直接看到探索质量
- 为后续事实/假设管理打基础

状态：执行中

### 第二批：事实与假设管理

目标：

- 让探索结果从“原始工具输出”升级为“结构化证据”

范围：

- 引入 `facts`
- 引入 `hypotheses_open`
- 引入 `hypotheses_rejected`
- 增加基础规则：
  - 没有足够事实，不能宣称根因
  - 有多个未排除假设时，只能输出“当前最可能”
- 将事实/假设摘要注入系统提示词

预期收益：

- 提高结论质量
- 降低来回推翻结论的问题

状态：待做

### 第三批：调用链与聚焦探索

目标：

- 让 agent 先定位调用链，再下钻实现细节

范围：

- 支持更强的模块入口摘要
- 支持从 `/scan` 继续下钻
- 支持更聚焦的符号范围探索
- 优先输出“调用链/依赖链/落点链”摘要

预期收益：

- 减少在实现细节之间无序横跳
- 提高复杂项目问题定位效率

状态：待做

### 第四批：语言与项目类型特化

目标：

- 针对 Python、Unity/C#、Node/TS 等主流项目给出更强探索策略

范围：

- 栈特化规则
- 项目类型特化入口识别
- 语言特定符号与结构摘要

预期收益：

- 在真实项目里获得更高的首轮定位成功率

状态：待做

## 第一批详细拆解

### 1. 结构化 investigation state

计划字段：

```json
{
  "investigation_focus": "",
  "files_read": [],
  "trees_viewed": [],
  "search_queries": [],
  "facts": [],
  "hypotheses_open": [],
  "hypotheses_rejected": [],
  "warnings": []
}
```

### 2. 重复探索告警

第一版规则保持简单：

- 同一路径被连续或高频重复读取
- 且读取之间没有新增事实
- 则生成 warning

warning 进入两处：

1. `/investigation`
2. 系统提示词

### 3. `/investigation`

用户可以直接查看：

- 当前 focus
- 最近扫描
- 最近符号查询
- 最近读取文件
- 已记录事实
- 当前 warning

---

## 验收标准

第一批完成后，应满足：

1. agent 连续多次读取同一文件时，session 中会留下 warning。
2. 系统提示词能看到该 warning。
3. `/investigation` 能把这些状态展示出来。
4. 新增能力不会破坏现有 `/scan` 和 `/symbols`。
