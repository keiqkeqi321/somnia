# 桌面端基础设施与 UI 实施计划

## 概述

本文定义 Somnia 在不影响现有 CLI 与核心能力的前提下，逐步演进到跨平台桌面应用的实施路径。

目标不是把当前 REPL 直接套进桌面壳，而是在现有工程之上先建设一层稳定的应用服务基础设施，然后基于这层基础设施实现另一套独立桌面 UI。

---

## 目标

- 保持现有 `somnia` CLI 用户体验与行为不回退
- 保持当前 runtime、存储结构、工具系统、权限模式、会话恢复逻辑继续可用
- 新增一层可复用的应用服务层，供 CLI 和桌面端共同消费
- 实现独立桌面 UI，而不是复用终端 UI
- 交付可安装的 Windows 与 macOS 桌面应用

## 非目标

- 不在第一阶段重写 runtime
- 不在第一阶段替换现有 CLI REPL
- 不在第一阶段做插件市场、多窗口、自动更新、远程同步等扩展能力

---

## 当前工程现状

### 已具备的能力

Somnia 当前核心能力已经集中在 `OpenAgentRuntime`，包括：

- provider 选择与切换
- session 创建、加载、列举与保存
- agent loop 与工具执行
- 背景任务、todo、MCP、team 协作
- 工具日志、上下文治理、回滚与 checkpoint

这说明 Somnia 的核心并不依赖 CLI 才能存在，runtime 具备继续抽象成应用服务层的基础。

### 当前主要耦合点

当前 CLI/REPL 与终端交互强耦合，主要体现在：

- `prompt_toolkit` 驱动输入框、补全与快捷键
- `print` / `input` 驱动输出与阻塞式选择
- 授权请求、模式切换请求、中断请求主要由 REPL runner 主线程协调
- 流式输出当前主要面向终端渲染而不是结构化事件

这意味着桌面端如果直接复用 REPL，会把终端交互模型原样搬进桌面壳，后续维护成本高，且难以形成稳定 API。

---

## 总体路线

采用“两层演进、双适配器”路线：

1. 保持现有 runtime 为唯一业务核心
2. 在 runtime 之上新增应用服务层
3. 让 CLI 成为应用服务层的一个适配器
4. 新增桌面 sidecar 与桌面 UI，作为应用服务层的另一组适配器

换句话说：

- runtime 负责能力
- app service 负责会话编排、事件流、交互请求抽象
- CLI 负责终端呈现
- Desktop UI 负责桌面呈现

---

## 设计原则

### CLI 零回归优先

任何桌面化改造都不能以破坏现有 CLI 行为为代价。

### 不复制业务逻辑

桌面端与 CLI 必须共用同一套 runtime 与服务层，避免形成两套执行逻辑。

### 先服务层，后 UI

只有先把交互、事件流和编排从终端里抽出来，桌面 UI 才不会变成“伪终端”。

### 存储兼容优先

继续沿用 `.open_somnia/` 下现有 session、transcript、task、team、jobs、permissions 等存储布局，桌面端不另起一套状态体系。

---

## 分阶段计划

## 阶段 0：建立桌面化基线

### 目标

冻结必须保持不变的 CLI 行为，并把后续演进的回归边界定义清楚。

### 工作项

- 梳理当前 CLI 与 runtime 的职责边界
- 明确必须保持不变的关键行为：
  - `somnia --workspace .`
  - `somnia -r`
  - 中断与排队
  - execution mode 切换
  - `request_authorization`
  - `request_mode_switch`
  - todo 状态展示
  - resume 过滤规则
- 将以下测试列为硬回归门槛：
  - `tests/test_cli_resume.py`
  - `tests/test_process_output.py`
  - `tests/test_repl_todo.py`
  - `tests/test_runtime_tool_output.py`

### 交付物

- 本文档
- 一份开发约束说明，可并入 `AGENTS.md` 或单独落文档

### 验收标准

- 后续任一阶段都不得绕过上述回归集

---

## 阶段 1：新增应用服务层

### 目标

在不改 CLI 用户可见行为的前提下，把 runtime 的可复用能力抽象成桌面端可消费的服务接口。

### 建议目录

```text
open_somnia/app_service/
  __init__.py
  models.py
  events.py
  session_service.py
  turn_service.py
  provider_service.py
  interaction_service.py
  runtime_host.py
```

### 核心职责

- 持有 `OpenAgentRuntime`
- 统一会话创建、加载、列举与切换
- 提供 `run_turn` 的结构化封装
- 把运行时过程转成结构化事件
- 抽象用户交互请求：
  - authorization request
  - mode switch request
  - interrupt request
- 抽象流式输出，不再依赖终端打印器

### 最小命令面

- `create_session`
- `list_sessions`
- `load_session`
- `run_turn`
- `interrupt_turn`
- `switch_provider_model`
- `set_reasoning_level`
- `recent_tool_logs`
- `render_tool_log`
- `list_providers`
- `list_models`

### 最小事件面

- `turn_started`
- `assistant_delta`
- `assistant_completed`
- `tool_started`
- `tool_finished`
- `authorization_requested`
- `mode_switch_requested`
- `interrupt_requested`
- `interrupt_completed`
- `session_updated`
- `todo_updated`
- `error`

### 交付物

- 可被终端与桌面共同消费的 Python 服务层
- 不依赖 `prompt_toolkit` 的 turn 执行入口

### 验收标准

- 不启动 REPL 也能通过服务层完整跑通一次会话
- 流式输出可通过事件流而不是 `print` 获得

---

## 阶段 2：将 CLI 改为服务层适配器

### 目标

保留现有 CLI 交互表现，但底层改为通过应用服务层执行。

### 工作项

- 保留 `open_somnia/cli/` 作为终端适配层
- 让 REPL runner 从“直接操纵 runtime”改为“消费 app service”
- 将以下终端专属逻辑保留在 CLI：
  - `prompt_toolkit`
  - `print` / `input`
  - 终端弹窗与交互选择
  - ANSI 渲染
- 将以下逻辑迁出 REPL：
  - turn 编排
  - 事件分发
  - 交互请求模型
  - 中断状态协调

### 交付物

- 基于 app service 的 CLI
- CLI 行为不变

### 验收标准

- 现有 CLI 关键测试继续通过
- 用户看不到行为差异

---

## 阶段 3：实现桌面 sidecar

### 目标

为桌面端提供一层本地进程服务，使 UI 与 Python runtime 解耦。

### 建议目录

```text
desktop/
  backend/
    __init__.py
    server.py
    ipc.py
    bootstrap.py
```

### 设计建议

- sidecar 使用 Python 运行
- sidecar 内部持有 `OpenAgentRuntime` 与 app service
- 前端与 sidecar 通过本地 IPC 通信

### 通信方式建议

优先方案：

- HTTP：处理命令型请求
- WebSocket：处理流式事件

备选方案：

- Tauri command + event bridge

优先选择前者的原因：

- 接口更清晰
- 调试成本更低
- 未来更易扩展 headless 调试与自动化测试

### 交付物

- 本地 sidecar 可单独启动
- UI 可通过 sidecar 进行会话、执行、打断、模型切换

### 验收标准

- 不依赖 CLI，sidecar 即可驱动一轮完整对话
- authorization / mode switch 请求可通过事件发出并等待外部响应

---

## 阶段 4：实现桌面 UI

### 目标

基于 sidecar 构建独立桌面 UI，而不是嵌入终端 UI。

### 技术建议

- 桌面壳：`Tauri 2`
- 前端：`React + TypeScript`

选择理由：

- 安装包与跨平台支持成熟
- UI 开发表达力高
- 前后端分层自然
- 比直接包装终端 UI 更容易做出可维护的桌面体验

### 第一版 UI 范围

- 会话列表
- 对话主窗口
- 流式输出
- 中断按钮
- provider / model 切换
- execution mode 展示
- authorization 弹窗
- mode switch 弹窗
- tool log 面板
- todo 面板
- resume 历史

### 暂缓项

- 多窗口
- 自动更新
- 富媒体拖拽大范围增强
- 插件市场
- 高级配置编辑器

### 交付物

- 独立桌面 UI
- 基于 sidecar 的完整核心交互闭环

### 验收标准

- 桌面端可覆盖 CLI 的常用交互主路径
- CLI 与桌面端共享同一工作区存储

## 阶段 5：安装器与分发

### 目标

交付用户可安装的 Windows 与 macOS 桌面应用。

### 平台目标

#### Windows

- 产物：`MSI`
- 首发优先：`x64`

#### macOS

- 产物：`.app + .dmg`
- 首发建议分别构建：
  - `arm64`
  - `x64`

如后续需要，再考虑 universal 包。

### 打包内容

- 桌面前端
- Python sidecar
- 必要运行时依赖
- 启动与更新元数据

### CI/CD 建议

- GitHub Actions matrix：
  - windows-latest
  - macos-latest
- 先做 unsigned internal build
- 再引入：
  - Windows 签名
  - macOS codesign
  - notarization

### 交付物

- Windows 安装器
- macOS 安装镜像
- 构建脚本与发布流水线

### 验收标准

- 用户无需预装 Python
- 安装后可直接启动桌面应用
- sidecar 能随安装包一起运行

## 风险与控制点

## 风险 1：为桌面端破坏 CLI 交互

控制方式：

- 所有基础设施改造以 CLI 零回归为最高优先级
- 阶段 2 之前不改现有用户可见交互

## 风险 2：把桌面端做成“终端套壳”

控制方式：

- 桌面端禁止直接复用 `run_repl`
- 必须先完成 app service 与 sidecar

## 风险 3：过早做安装器，拖慢研发

控制方式：

- 安装器放在 UI 与 sidecar 跑通之后
- 阶段 5 之前只做开发态启动链路

## 风险 4：形成两套逻辑

控制方式：

- CLI 与桌面端只允许通过 app service 使用 runtime
- 不允许桌面端私接 runtime 内部细节

完成阶段 0 与阶段 1，服务层可独立跑通。

### M2

完成阶段 2，CLI 切换到服务层，行为无回归。

### M3

完成阶段 3，sidecar 可独立提供会话与事件服务。

### M4

## 推荐的下一步

第一步不要先做 UI，而是先把阶段 1 细化成一份可执行的接口设计文档，至少明确：

- app service 的模块边界
- 命令模型
- 事件模型
- authorization / mode switch / interrupt 的响应机制
- CLI 与桌面端各自消费方式

只有这层边界稳定后，桌面端 UI 与安装器工作才不会反复返工。
