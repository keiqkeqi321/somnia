# 权限系统与执行模式

## 概述

Somnia 的权限系统通过 **Execution Mode（执行模式）** + **PermissionManager（权限管理器）** 双层机制控制 Agent 的工具访问权限，确保在不同场景下 Agent 具有合适的能力范围。

---

## 执行模式

### 四种模式

| 模式 | Badge | 说明 | 文件编辑 | 任务管理 | 团队协作 |
|------|-------|------|----------|----------|----------|
| `shortcuts` | ❓ | 仅快捷查询 | ❌ | ❌ | ❌ |
| `plan` | ⏸ | 规划模式 | ❌ | ❌ | ❌ |
| `accept_edits` | ⏵⏵ | 接受编辑 | ✅ | ✅ | ✅ |
| `yolo` | ❗ | 完全自主 | ✅ | ✅ | ✅ |

### 模式指引

每个模式在系统提示中注入对应的指引文本，告知 Agent 当前的能力边界。

#### shortcuts（快捷模式）
- 仅使用只读工具
- 需要阻断操作时调用 `request_authorization`
- 进入规划或实现阶段时调用 `request_mode_switch`

#### plan（规划模式）
- 保持只读
- 返回具体实现计划后再请求更高权限
- 用户确认实施后切换到 `accept_edits`

#### accept_edits（接受编辑模式，默认）
- 可以编辑文件（`write_file` / `edit_file`）
- 可以创建、更新、认领任务
- 可以使用团队协作工具
- 其他阻断工具需 `request_authorization`

#### yolo（完全自主模式）
- 完全自主，无权限限制
- 仍应避免破坏性操作

### 模式切换

```
request_mode_switch(target_mode, reason)
        │
        ├── 降级切换（如 accept_edits → plan）
        │       └── 直接批准
        │
        └── 升级切换（如 plan → accept_edits）
                ├── 有 handler → 交互式用户审批
                └── 无 handler → 拒绝
```

### 工具分类与模式限制

| 工具分类 | 包含工具 | shortcuts | plan | accept_edits | yolo |
|----------|---------|-----------|------|-------------|------|
| 只读工具 | `project_scan`, `tree`, `find_symbol`, `glob`, `grep`, `read_file`, `load_skill`, `compress`, `TodoWrite`, `task_get`, `task_list`, `list_teammates`, `check_background` | ✅ | ✅ | ✅ | ✅ |
| 文件编辑 | `write_file`, `edit_file` | ❌ | ❌ | ✅ | ✅ |
| 任务变更 | `task_create`, `task_update`, `claim_task` | ❌ | ❌ | ✅ | ✅ |
| 团队协作 | `spawn_teammate`, `send_message`, `read_inbox`, `broadcast`, `shutdown_request`, `plan_approval` | ❌ | ❌ | ✅ | ✅ |
| 权限/模式 | `request_authorization`, `request_mode_switch` | ✅ | ✅ | ✅ | ✅ |

## 权限管理器

### PermissionManager 三层授权

```
工具调用
    │
    ▼
┌──────────────────────────────────────┐
│  Layer 1: Workspace 授权             │
│  permissions.json 中持久化授权的工具  │
│  → 自动放行                           │
└──────────────┬───────────────────────┘
               │ 未授权
               ▼
┌──────────────────────────────────────┐
│  Layer 2: 一次性授权                 │
│  通过 request_authorization 批准     │
│  scope=once → 放行 1 次后失效        │
│  scope=workspace → 永久授权           │
└──────────────┬───────────────────────┘
               │ 未授权
               ▼
┌──────────────────────────────────────┐
│  Layer 3: 执行模式检查               │
│  根据当前模式决定是否阻断            │
│  阻断 → 返回友好的阻断消息            │
└──────────────────────────────────────┘
```

### 授权持久化

工作区授权存储在：
```
.open_somnia/data/permissions.json
```

格式：
```json
{
    "authorized_tools": ["bash", "write_file", "edit_file"]
}
```

### Subagent 特殊规则

| 执行模式 | Explore subagent | general-purpose subagent |
|----------|-----------------|------------------------|
| `accept_edits` / `yolo` | ✅ 允许 | ✅ 允许 |
| `shortcuts` / `plan` | ❌ 需 request_authorization | ❌ 需用 Explore 或切换模式 |

### 请求授权工具

```python
request_authorization(tool_name, reason, argument_summary)
```

返回结果：
```json
{
    "status": "approved",    // approved | denied
    "scope": "workspace",    // workspace | once
    "tool_name": "bash",
    "reason": "需要编译项目"
}
```

## 权限检查流程

```python
# ToolRegistry.execute() 中
authorizer = runtime.authorize_tool_call
blocked = authorizer(tool_name, payload, ctx=ctx)
if blocked is not None:
    return blocked  # 阻断消息
return tool.handler(ctx, payload)  # 执行
```
