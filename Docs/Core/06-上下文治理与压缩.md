# 上下文治理与压缩

## 概述

Somnia 的上下文治理分三层：

1. **Payload Normalization**：只对发给模型的 payload 副本做确定性归一化
2. **Semantic Janitor**：对历史工具结果做语义级脱水
3. **Auto Compact**：在高占用时把长历史压成 continuation summary

另有一条旁路信号 **Topic Shift Assist**，只用于帮助 runtime 更早判断是否值得触发 janitor，不直接改写主会话。

---

## 当前流程

1. 用户消息先写入 `session.messages`
2. runtime 在用户发问边界判断是否需要触发 Semantic Janitor
3. `build_payload_messages()` 构造 payload 副本并做轻量归一化
4. 如果上下文进一步逼近上限，则执行 Auto Compact

设计原则：

- payload 层只做轻量、确定性、payload-only 的处理
- 真正的体积治理交给 Semantic Janitor 和 Auto Compact
- 恢复原文依赖 `log_id` 和 tool log，而不是依赖 payload 副本

---

## 第一层：Payload Normalization

入口：`open_somnia/runtime/compact.py` -> `build_payload_messages(...)`

它只做四件事：

1. 深拷贝消息，避免改写原始 `session.messages`
2. 从 payload 副本里的 `tool_result` 移除 `raw_output` / `log_id`
3. 对较大的完全重复工具结果做 exact dedupe
4. 对 `read_file` 做同文件区间重叠抑制

### Exact Dedupe

完全重复去重是确定性的，不做语义判断。当前主要基于：

- `tool_name`
- 归一化后的 `tool_input`，忽略 `importance`
- `content` 哈希

旧副本会被替换成简短占位，例如：

```text
[Duplicate tool result omitted | read_file] Identical output appears later.
```

### `read_file` 同文件区间重叠抑制

这一步也是 payload-only，但触发范围更窄：

- 只有**最新一轮** `tool_result` 里出现了 `read_file`，才会触发
- 触发后只处理**这一轮读过的路径**
- 按同路径、从新到旧逆向扫描
- 后读到的更聚焦结果优先保留

行为：

- 更早结果若被完全覆盖，会替换成占位：

```text
[Overlapping read_file result omitted | demo.txt:3-8] Covered by later read(s) of the same file.
```

- 更早结果若只部分重叠，只裁掉重叠行，保留前后独有内容，并插入显式 overlap marker
- 带字符截断标记的 `read_file` 结果、Semantic Janitor 摘要、`evicted` 占位、exact dedupe 占位都不会参与区间裁剪

### 保证

- 不改写原始 `session.messages`
- 不影响 transcript
- 不影响 `request_original_context(log_id)` 的恢复能力
- 没有本轮 `read_file` 时，这一步几乎没有额外成本

## 第二层：Semantic Janitor

阈值：默认读取 `runtime.janitor_trigger_ratio = 0.6`

```toml
[runtime]
janitor_trigger_ratio = 0.6
```

### 执行时机

- 只在**用户发问边界**触发
- 不在 `_messages_for_model()` 中途触发
- 不在 `context_window_usage()` 内触发
- 保护最近 2 轮工具结果不被处理

### 做什么

Semantic Janitor 会直接回写历史 `tool_result`：

- 保留 `log_id`
- 移除 `raw_output`
- 把 `content` 改成摘要或驱逐提示

它的目标不是只让“本轮 payload 变小”，而是让后续轮次真正瘦身，减少重复深拷贝、重复计数和重复 I/O。

### Topic Shift Assist

这是 janitor 的旁路信号，不是正式会话消息：

- 不写入 `session.messages`
- 不写入 transcript
- 只返回治理信号，如 `context_shift: true/false`

触发规则：

- `ctx < 20%`：不做切题检测
- `20% <= ctx < 60%`：只在用户发问边界允许旁路检测
- `ctx >= 60%`：仍以自动 janitor 规则为主，Topic Shift 只作为额外信号

### 工具结果状态

| 状态 | 行为 | 适用场景 |
|------|------|----------|
| `original` | 保持完整 | 当前文件、关键报错、强相关证据 |
| `condensed` | 替换为 1 到 2 句事实摘要 | 早期探索结果、已吸收结论 |
| `evicted` | 只保留恢复提示 | 重复目录浏览、一次性确认 |

### 判断信号

当前主要参考：

- 最近 2 到 4 轮 user / assistant 文本
- 当前关注文件、符号、错误关键词
- 工具类型
- 输出长度
- recency
- todo 的 `in_progress`
- `importance`

其中 `importance` 不是单独裁决条件，只是参与排序和加权。

### 低收益熔断

自动 janitor 会记录一次脱水比例：

```text
dehydration_ratio = (before_tokens - after_tokens) / before_tokens
```

规则：

- 只看**最近一次自动 janitor**
- 如果 `dehydration_ratio < 10%`，本会话后续自动 janitor 直接熔断
- 手动 `/janitor` 不受这个熔断影响

### 持久化后的表现

janitor 处理过的 `tool_result` 会表现为：

- `semantic_state` 写回消息项
- `content` 变成摘要或驱逐提示
- `raw_output` 被移除
- `log_id` 保留

## 第三层：Auto Compact

阈值：`AUTO_COMPACT_TRIGGER_RATIO = 0.82`

入口：`open_somnia/runtime/compact.py` -> `CompactManager.auto_compact(...)`

流程：

1. 先把完整会话写入 transcript snapshot
2. 调模型生成 continuation summary
3. 用两条压缩消息替换原始长历史
4. 重新计算 recent context snapshot，避免 UI 继续显示旧占用

摘要必须保留：

- Current goal
- Confirmed decisions
- Open work
- Files changed
- Constraints
- Risks

压缩后的格式是：

```text
[compacted context]
{continuation summary}

Understood. Continuing from compacted context.
```

---

## 恢复原始上下文

工具：`request_original_context(log_id)`

当 janitor 将工具结果压缩或驱逐后，Agent 仍可通过 `log_id` 恢复完整原始输出：

```text
[Restored tool output | bash | log abc123]
<full output here>
```

恢复原则：

1. 按 `log_id` 精确定位
2. 内容直接来自 `.open_somnia/logs/tool_logs/{log_id}.json`
3. 恢复动作是显式的，不自动回填历史消息

---

## 手动命令

- `/janitor`：手动执行语义清理
- `compress`：执行整体 compact

区别：

- `janitor` 只处理历史工具结果，尽量保留原会话结构
- `compact` 会把长历史折叠成 continuation summary

---

## 当前边界

- payload 层不会做全局历史扫描，只做轻量、局部、确定性的归一化
- `read_file` 重叠抑制不是全局清理器；某个历史文件如果本轮没有再次读取，它不会主动介入
- 真正的跨轮体积治理仍主要依赖 Semantic Janitor 和 Auto Compact
- 自动 janitor 不在 `context_window_usage()` 内触发，REPL 状态栏只读取最近快照，避免 UI 重绘反复进入完整上下文计算链
