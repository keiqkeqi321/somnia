# 探索能力

## 概述

Somnia 的探索能力提供即时探索工具，帮助 Agent 快速建立项目地图、定位符号和读取源码。当前仅保留即时探索，**不再内置**探索结果持久化、repo summary、session 级 exploration cache，也不再提供 `/investigation` 面板。

探索阶段的原则：

1. 先建立局部项目地图。
2. 再定位符号和落点。
3. 最后读取最少量的必要源码。

---

## 当前可用能力

### 1. `/scan`（项目扫描）

底层工具：`project_scan`（`open_somnia/tools/filesystem.py`）

用途：

- 快速扫描当前仓库或某个子目录
- 输出项目根、文件/目录数量、可能技术栈、指导文件、源码根和浅层目录树
- 作为开始探索时的第一步，帮助 agent 建立项目地图

用法：

```text
/scan
/scan src
/scan --refresh src
```

说明：

- `/scan` 默认扫描工作区根目录
- `/scan some/path` 扫描指定子目录
- `--refresh` 仅为兼容保留参数，不再涉及任何缓存刷新
- `/scan` 每次都是即时查询，不会写入 repo summary，也不会复用旧扫描结果

### 2. `/symbols`（符号查找）

底层工具：`find_symbol`（`open_somnia/tools/filesystem.py`）

用途：

- 按名称子串查找类、函数、方法、接口等符号
- 在真正读文件前，先缩小实现范围
- 结合交互选择器快速预览命中的源码位置

用法：

```text
/symbols
/symbols build
/symbols PaperComponent
```

说明：

- 不带查询词时弹出输入框
- 找到多个结果后，弹窗让用户选择一个结果进行源码预览
- `/symbols` 也是即时查询，不会把结果沉淀成 session 记忆

### 3. 其他探索工具

以下工具可供 agent 在运行时使用：

| 工具 | 说明 |
|------|------|
| `tree` | 浅层目录树渲染 |
| `glob` | 文件模式匹配搜索 |
| `grep` | 文件内容搜索 |
| `read_file` | 读取文件内容 |

推荐探索顺序：

1. `project_scan` → 建立项目地图
2. `tree` → 聚焦子目录
3. `find_symbol` → 定位符号实现
4. `read_file` → 读取必要源码

---

## 为什么移除了探索记忆

上一版探索能力把扫描摘要、符号查询、文件读取等状态写进了 session 和工作区存储，还尝试把这些内容重新注入系统提示。

这套设计已删除，原因：

- 记忆质量不稳定，容易把过时结论带回后续轮次
- "读过"不等于"验证过"，会放大错误先验
- 探索输出和真正可复用知识没有被严格区分
- 记忆层和压缩层耦合后，排查行为退化更难定位

---

## 当前边界

- 不会持久化探索摘要
- 不会维护 `.open_somnia/repo_summary.json`
- 不会在 session 里写 `exploration_cache`
- 不会提供 `/investigation`
- 需要保留结论时，通过正常对话、任务系统或后续独立记忆系统处理

---

## 优化方向

后续探索能力优化分三条独立路线：

### 1. 即时探索质量优化

- 更强的源码根识别
- 更好的符号结果排序
- 更适合大仓库的扫描摘要
- 对常见项目类型增加更准确的入口提示

### 2. 结构化证据层

- 让 agent 在当前任务中显式整理"事实、假设、待验证项"
- 内容仅属于当前推理链，不自动持久化为长期记忆
- 与压缩策略解耦，避免互相污染

### 3. 新记忆系统（独立设计）

- 记忆来源必须可追溯
- 记忆项必须允许失效和清理
- 记忆和即时探索分层
- 不能把未经验证的探索输出当成稳定事实
